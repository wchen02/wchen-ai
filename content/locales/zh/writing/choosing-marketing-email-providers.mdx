给 newsletter 或营销 drip 选邮件服务商，听起来应该很简单：发邮件、收订阅、也许统计打开。市场很挤：Buttondown、Resend、ConvertKit、Mailgun 等一堆都说能搞定。

我认真比了四个——Buttondown、Resend、ConvertKit、Mailgun——从「开发者主导、API 优先」的工作流出发。下面是对我来说有意义的对比，以及为什么最后选了 Resend。

## 对比

|                        | **Buttondown**              | **Resend**                                | **ConvertKit**                    | **Mailgun**                               |
|------------------------|-----------------------------|--------------------------------------------|-----------------------------------|--------------------------------------------|
| **主要定位**           | Newsletter、写作者          | 事务 + 营销，开发者                        | 创作者、营销人                    | 事务 + 营销，企业                          |
| **API / 开发体验**     | REST API，文档还行          | API 优先，SDK（Node、React），文档很好     | REST API，偏创作者                | 成熟 REST API，配置较重                    |
| **定价模式**           | 按订阅数                    | 按邮件量（免费档 3k/月）                    | 按订阅数，免费至 1k               | 按邮件量，按量档位                          |
| **代码 / 自动化**      | 简单自动化够用              | 为 pipeline、React Email、webhook 设计     | 表单 + UI 内自动化                | 完全可控，运维负担更大                      |
| **送达率**             | 做 newsletter 够稳         | 强，基础设施新                             | 做创作者漏斗强                    | 企业级                                     |
| **最适合**             | 单人写作者、最少配置        | 开发者、应用、程序化发送                    | 非技术创作者、漏斗                | 高量、已有 Mailgun 用户                     |

## Buttondown

**优点：** 直接。就是一个能用的 newsletter 产品：订阅表单、草稿、排期、简单 API，没有多余东西。价格随订阅数走，可预期。想用 Markdown 写、不发愁送达和模板的话很适合。

**缺点：** 产品围绕 dashboard 和 newsletter 流程设计，不是以*你的代码*为真相源。API 能干活但并非设计中心。要从自己应用里管分段、触发、复杂流程会有点逆着产品来。自定义域名和品牌有，但相对「盒子里的写作者」体验是次要的。

## Resend

**优点：** 为开发者做。API 优先意味着所有能力都能从代码触达，没有「只能在 UI 里」的缺口。Node 和 React（含 React Email）的 SDK 让在代码里组邮件、一发即送很简单。Webhook、域名、审计日志都是一等公民。免费档（每月 3,000 封）足够上线和迭代。文档清楚，心智模型是「从应用发邮件」，不是「管一个独立的邮件产品」。

**缺点：** 比 Mailgun、ConvertKit 年轻。预制模板少，没有内置可视化活动编辑器；要拖拽做活动就得用 React Email 或自己搭。产品为「习惯写代码」的人优化。

## ConvertKit

**优点：** 对创作者和营销人很强。表单、落地页、标签、自动化、漏斗都在产品里，送达针对 newsletter 和营销调优。如果你想在 UI 里设计序列和分段、偶尔碰一下 API，ConvertKit 很合适。

**缺点：** 重心在 UI 不在 API。自动化和逻辑都在他们的 builder 里；全在代码里做得到但不是默认路径。对想完全在代码里拥有流程——注册、分段、触发——的开发者，抽象会有点倒挂。定价按订阅数，所以列表大小比发送量更驱动成本。

## Mailgun

**优点：** 成熟、稳定、功能全。这个站收信和路由我用它；API 和文档可预期，送达和合规工具都是企业级。需要高量、详细日志和完全控制时很合适。

**缺点：** 定位和体验更偏「邮件基础设施」而不是「开发者友好产品」。配置和概念（域名、路由、webhook）强大但更重。纯外发营销或 newsletter 能做，但不像 Resend、ConvertKit 那样为那类用例优化。换来的是灵活和控制，代价是配置和认知负担。

## 为什么选 Resend

我要的是*开发者友好*且*API 优先*的供应商。

对我而言开发者友好就是：注册、加域名、从应用里发信几分钟搞定；文档直接回答「用代码怎么做 X」而不是把我赶去 dashboard；错误和 webhook 为调试设计。Resend 的 SDK 和 React Email 集成符合这点，设计和发活动不用离开编辑器。

API 优先意味着产品不假设工作发生在 dashboard。订阅者、事件、发送都能通过 API 操作。我可以在自己的代码里做注册流程、分段和触发，把 Resend 当投递层。这和「基础设施最好别挡道」是同一逻辑：最好的工具是贯彻你意图、不强行换工作流的那种。

如果你的工作流是「在它们 UI 里管列表和活动」，Buttondown 和 ConvertKit 更合适。如果已经用 Mailgun 做事务邮件或要最大控制和量，Mailgun 更合适。做应用或开发者主导的 newsletter、代码为真相源时，Resend 是最顺的。

选邮件服务商在大局里是个小决定——但每次加注册表单、每次发活动都会碰到。选一个和你工作方式匹配的，每次「不用再开一个标签就能做成一次函数调用」都在回本。
